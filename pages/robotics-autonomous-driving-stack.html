Part 01 · 全局层

先建立分层心智模型

一句话定位五个工具，再用一张"责任分层栈"看清谁负责哪一层——这是后面所有细节的地图。

🧠
一句话心智模型

ROS1/ROS2 是神经系统（模块间怎么通信）；Simulink 是算法图纸 + 代码生成机（控制/逻辑怎么设计）；CarSim 是车的身体物理（车怎么动）；CARLA 是世界与感官（车看到什么、世界长什么样）。而路径规划算法，是跑在这套系统里的大脑决策逻辑。

很多人把这几个名字摆在一起会觉得"它们是不是竞品、该选哪个"。其实它们处在完全不同的层——一个真实项目里，它们往往是同时存在、各司其职的。理解了"谁负责哪一层"，整个自动驾驶 / 机器人的部署链路就清晰了。

一、谁负责哪一层

把一辆自动驾驶车（或一台移动机器人）的"软件—物理"栈竖起来看，五个工具刚好各占一层，而 ROS 像一根总线把它们串起来：

工具本质是什么所在层一句话作用
ROS1通信中间件 + 工具生态软件框架让感知/规划/控制等模块解耦地互相收发消息
ROS2ROS1 的工业级重写（基于 DDS）软件框架同上，但支持实时、多机、安全，可量产

CarSim高保真车辆动力学仿真器物理 / 被控对象精确算出"给定油门转向，车实际怎么动"
CARLA开源自动驾驶世界仿真器(Unreal)环境 / 传感器提供地图、交通、天气与各类传感器数据流

二、ROS1 与 ROS2：同一思想的两代

它们不是两个东西，而是同一套理念的两代实现。核心都是把系统拆成一个个 节点(node)，节点之间用 话题(topic) 发布/订阅消息、用 服务(service) 做请求应答。区别在"工程成熟度"：

维度ROS 1ROS 2
发现机制中心化：需要 roscore/Master，单点故障去中心化：基于 DDS，无 Master
通信底层自研 TCPROS / UDPROSDDS（Fast-DDS / Cyclone DDS）
实时性几乎没有支持（配合 RTOS / QoS 策略）
QoS 配置无可靠性 / 时延 / 历史深度等可调
多机器人 / 安全弱原生多机 + SROS2 安全
定位研究 / 原型面向产品落地
现状最后一版 Noetic，2025 年已 EOL主流（Humble / Jazzy 等 LTS）
✅
怎么选

新项目直接上 ROS2；只有维护老代码才碰 ROS1。两者可通过 ros1_bridge 互通，迁移期常见两边并存。

三、Simulink / CarSim / CARLA 各自管什么

三者都跟"仿真"沾边，但保真维度完全不同——这正是它们经常被组合使用的原因：保真度要按需分配，你不会用 CARLA 算轮胎滑移，也不会用 CarSim 渲染相机图像。

Simulink —— 控制算法的图纸与代码生成机

  • MATLAB 家族里的图形化建模工具，用框图搭控制律、状态机、信号流。
  • 核心价值是 Model-Based Design：先在框图里把算法验对，再用 Embedded Coder 一键生成 C/C++ 烧到 ECU——这是汽车行业（AUTOSAR）标准做法。
  • 配套 Automated Driving Toolbox、ROS Toolbox（能直接生成 ROS/ROS2 节点）。

CarSim —— 车辆动力学的"数字替身"

  • 专做高保真车辆动力学：轮胎力学、悬架、转向、传动、整车质量分布……给定油门/刹车/转向，精确算出车的六自由度运动。
  • 它扮演被控对象（plant）：上层给指令，CarSim 回报"车实际怎么动"。研究 ESC / ABS / AEB / 操稳性时是金标准。
  • 同门兄弟：TruckSim（卡车）、BikeSim（摩托）。最经典的是 Simulink + CarSim 联合仿真。

CARLA —— 世界与传感器的开源仿真场

  • 基于虚幻引擎(Unreal)的开源自动驾驶仿真器，强项是感知与场景：高精地图、天气光照、车流人流，以及相机/LiDAR/Radar/IMU/GNSS 等传感器数据。
  • 提供 Python API 与 ROS bridge，是感知、规划、决策、强化学习训练的常用试验台。
  • 短板：自带车辆物理（PhysX）相对简化。要精细动力学时，可让 CARLA 出场景、把车辆模型交给 CarSim（CARLA 官方支持 CarSim co-sim）。

四、Simulink 深入：模块化的"控制器 + 被控对象"

Simulink 由 MathWorks 于 1984 年推出，是架在 MATLAB 之上的图形化建模与仿真环境。它的世界观非常简单，只有两类东西：模块(block) 和连接它们的信号线(line)。你把系统画成框图，Simulink 用求解器(solver) 按时间步推进、解微分方程，得到系统随时间的响应。

五个核心概念

概念是什么为什么重要
Block 模块一个运算单元：增益、积分、传递函数、查表、子系统…系统的"零件"
Signal 信号线模块间流动的数据（标量 / 向量 / 总线）系统的"管路"
Subsystem 子系统把一组模块打包成一个块，可层层嵌套模块化与复用的基础
Solver 求解器定步长 / 变步长，决定数值积分方式实时 / 嵌入式系统必须用定步长
Stateflow状态机 / 流程图插件做挡位、模式切换等决策逻辑

最经典的范式：闭环"控制器 + 被控对象"

Simulink 的招牌用法，是把控制器(Controller) 和被控对象(Plant) 画在同一张图里一起仿真——期望值减去反馈得到误差，控制器算出控制量驱动被控对象，输出再反馈回来，形成闭环：

真正的杀手锏：Model-Based Design（MBD）

Simulink 不只是用来仿真，更是一条"一个模型从设计用到量产"的流水线。先在桌面把"控制器 + 被控对象"一起验对（MIL），再用 Simulink Coder / Embedded Coder 自动生成 C/C++ 代码，逐级 SIL/PIL/HIL，最后烧进 ECU。这是汽车（AUTOSAR）、航空航天的标准开发范式：

这是最容易混的一对。两个都叫"仿真器"，但仿真从来不是一个东西——它是把世界、车、传感器、算法分别做成"替身"。CARLA 和 CarSim 恰好是其中两个不同的替身。用拍电影打比方最直观：

一句话角色

CARLA 是摄影棚 + 摄像机 + 群众演员——搭场景、出画面、布置车流人流；CarSim 是主角那辆车的特技替身——精确还原车身在物理世界里怎么动。一个管"世界与感官"，一个管"车辆动力学"，两者在"自车"这一点对接。

在一次完整闭环仿真里，它们这样分工

  • CARLA 负责"自车看到什么世界"——渲染传感器数据、生成场景与交通，构成环境与感官闭环。
  • 中间是你的规划 / 控制算法——把感知结果算成转向 / 油门指令。
  • CarSim 负责"自车听到指令后怎么动"——精确算出车辆六自由度运动，构成车辆动力学闭环，再把位姿回灌 CARLA。
🎚️
按需选三档保真

只测感知 / 规划 → 纯 CARLA（它自带的简化车辆就够）；只测控制 / 底盘 → 纯 Simulink + CarSim（根本不需要渲染画面）；端到端、既要真实感知又要真实动力学 → CARLA + CarSim 联合仿真。保真度越高越慢越贵，别为用不到的精度买单。

回看图 4 的联合仿真数据流：CARLA 在最上面出"世界"，CarSim 在最下面出"车身"，中间靠 ROS2 与规划/控制把它们焊成一个闭环——这就是两者在仿真里的全部关系。

七、贯穿部署的主线：X-in-the-Loop

这才是把五个工具统一起来的真正逻辑——它们对应开发 V 模型里不同阶段的测试。理解了这条主线，你就知道每个工具"什么时候上场"，以及为什么仿真做得越靠后越接近真实、成本越高。

阶段被测对象典型工具组合保真 / 成本
MIL
Model-in-Loop		算法模型本身Simulink(控制器) + CarSim/CARLA(被控对象)低 · 快 · 便宜
SIL
Software-in-Loop		生成的 C/C++ 代码编译后软件 + CARLA/CarSim，ROS2 跑节点↑
HIL
Hardware-in-Loop		真实 ECU / 控制器硬件实体控制器 + 实时仿真器(CarSim RT)↑↑
VIL
Vehicle-in-Loop		真车 + 虚拟环境真车在场地 + CARLA 注入虚拟交通↑↑↑
真实部署整车 / 整机ROS2 跑在车载计算单元上高 · 慢 · 贵
🛣️
部署流水线一句话

在仿真里（CARLA/CarSim/Simulink）把算法验到足够安全 → 生成可执行代码 → 用 ROS2 封装成节点 → 逐级 MIL→SIL→HIL→VIL → 上真车。同一套 ROS 软件架构从仿真一直用到量产，这就是 ROS 最大的价值。

运动规划层的算法虽多，但底层思路就三类：在格子上搜、在空间里采样、用曲线/优化算。一张图看懂区别：

🔑
A* 的灵魂：一个代价函数

几乎所有搜索型规划都绕不开这条式子——把"已走代价"和"到目标的乐观估计"加起来，优先扩展最有希望的节点：

A* 评估函数
\[ f(n) = \underbrace{g(n)}_{\text{起点→}n\text{ 的真实代价}} + \underbrace{h(n)}_{\text{ }n\text{→目标的启发估计}} \]

当 \(h(n)=0\) 时它退化成 Dijkstra（无方向、慢但最优）；当 \(h(n)\) 是"不高估"的乐观估计（如直线距离）时，A* 既快又能保证最优。这就是搜索型算法的核心直觉。

十、模块化设计落到非结构化环境

前面"感知 → 规划 → 控制"的模块化流水线，是在结构化道路（高速、城区，有车道线、有高精地图）下最顺手的。一旦进入非结构化环境（越野、矿区、港口、园区、农田、星球表面），这条流水线每一层都要换打法——因为 Frenet / Lattice 这些主力算法，骨子里都依赖"一条参考线 + 一张高精地图"，而非结构化环境恰恰两样都没有。

维度结构化（高速 / 城区）非结构化（越野 / 矿区 / 园区）
先验地图车道级高精地图无 / 只有粗略地形图
参考线车道中心线（现成）没有，需自己生成
感知重点车道线、交通参与者、信号灯可通行性、坡度、松软地面、负障碍（坑）
环境表示车道图 + 物体列表栅格代价图 / 高程图 / 可通行图
规划主力Frenet、Lattice、行为决策Hybrid A*、RRT*、costmap 局部规划
典型速度高低 — 中
典型场景Robotaxi、高速领航(NOA)矿卡、港口 AGV、农机、巡检 / 越野机器人、火星车

模块化流水线具体怎么改

  • 感知：从"找车道线"变成"估可通行区域"——点云地面分割、高程图 / 表面法向、可通行性语义分割、负障碍检测。
  • 定位：高精地图不可用时，更依赖 LiDAR / 视觉 SLAM、RTK-GNSS、轮速里程计的融合。
  • 规划：丢掉 Frenet 的参考线假设，回到栅格 / 采样——用 Hybrid A*（满足最小转弯半径）做全局，配 costmap 局部规划（如 Nav2），把地形代价直接灌进 costmap。
  • 控制：低速但需考虑轮地交互与打滑，常用 MPC。
🧱
为什么越野/矿区更像"移动机器人"而非"自动驾驶"

这正是为什么非结构化自动驾驶大量沿用移动机器人那套 costmap + Hybrid A* + DWA/TEB（即 ROS Nav2），而不是直接套 Apollo / Autoware 的车道级方案——两者对环境的根本假设不同：一个假设"有路有图"，一个假设"无路自己探"。

十一、路径规划算法清单

A · 基于搜索（有地图、结构化）

算法核心思想特点 / 场景
Dijkstra无启发的最短路保证最优但慢；路网路由

Dubins / Reeds-Shepp车类最短曲线（含倒车）泊车、低速调头
MPC滚动时域优化，预测 + 约束规划与控制通吃，工业落地热门
CHOMP / STOMP / TrajOpt轨迹优化机械臂运动规划

D · 局部反应式（实时避障，移动机器人常用）

算法核心思想场景
DWA 动态窗口在速度空间采样选最优ROS Nav 默认局部规划器

E · 学习类（前沿）

  • 强化学习(DRL)：端到端决策或局部规划，仿真训练为主——CARLA 是热门训练场。
  • 模仿学习(IL)：从人类驾驶数据学策略。
  • 神经运动规划：用网络加速采样 / 搜索（如学习采样分布）。

十三、Autoware 框架剖析

把前面所有拼图拼成一个真实工程，最好的样板就是 Autoware——它几乎用上了本页讲到的每一个概念：ROS2 总线 + 分层规划 + 可对接 CARLA 仿真。

Autoware 源自日本（2015 年由名古屋大学加藤真平发起），现由 Autoware Foundation + TIER IV 主导，以 Apache-2.0 开源，是跑在 ROS 上最完整的开源自动驾驶全栈，已被丰田 e-Palette、Foxconn 等采用。

三代演进

  • Autoware.AI：初代，基于 ROS1，已停止维护。
  • Autoware.Auto：ROS2 上的干净重写，强调代码质量与测试。
  • Autoware Universe / Core：当前主线，ROS2，模块最全、社区最活跃。

七大模块栈：感知 → 决策 → 执行

模块干什么代表方法
Sensing采集并预处理传感器LiDAR / 相机 / GNSS / IMU 驱动与同步
Map提供先验地图Lanelet2 矢量高精地图 + 点云地图
Localization估自车位姿NDT 点云配准 + EKF 融合
Perception感知动态环境检测、跟踪、轨迹预测
Planning生成轨迹路由(Lanelet2) → 行为 → 运动(避障 / 优化)
Control跟踪轨迹MPC / Pure Pursuit trajectory follower
Vehicle I/F对接线控底盘转向 / 油门 / 刹车指令与状态回报
🧩
Autoware = 本页所有概念的合订本

Autoware 本身就是一堆 ROS2 节点；它的 Planning 正是第八节那套路由→行为→运动分层；它能通过 carla_autoware bridge 直接接 CARLA 做闭环仿真。换句话说：ROS2 + 分层规划 + CARLA 仿真这三块拼图，在 Autoware 里拼成了一台真车能跑的系统。

顺带一提与 Apollo 的区别：两者都是开源全栈；Autoware 基于 ROS2、由全球学术 / 工业社区驱动、用 Lanelet2 地图；Apollo 由百度主导，自带更完整的云端工具链与数据闭环。

十四、统一理解：一条完整链路

把全部串起来，一个典型自动驾驶 / 机器人项目长这样：

  1. ROS2 是骨架——所有模块按发布订阅组织成节点，仿真和真车共用同一套软件架构。
  2. CARLA 在开发期提供"世界 + 感官"，喂感知数据、测规划决策，还能当 RL 训练场和场景回归测试台。
  3. 路径规划算法（A* / RRT* / Frenet / MPC……）作为规划节点跑在 ROS2 里，分四层调用。
  4. Simulink 设计并验证控制器，生成 C/C++ 部署到车载 ECU，并接回 ROS2。
  5. CarSim 在控制 / 动力学验证时充当高保真车辆模型，配合 Simulink 做 MIL / HIL。
  6. 整个过程沿 MIL → SIL → HIL → VIL → 真车 逐级放行。
它们不是竞品，而是一条流水线上的不同工位：CARLA 管"看见"，规划算法管"想清楚"，Simulink/CarSim 管"开起来"，ROS 把它们焊成一台能从仿真走到量产的机器。
